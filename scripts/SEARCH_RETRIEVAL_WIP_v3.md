## 1. 서비스 컨셉: 왜 검색 정확도가 중요한가

### 1.1 메디스캐너란?

**메디스캐너(MediScanner)**는 **개인 맞춤형 의료 질의 서비스**입니다.

일반적인 의료 챗봇과 다른 점:

| 일반 의료 챗봇 | 메디스캐너 |
|--------------|-----------|
| "고혈압 치료법은?" → 일반적인 답변 | "고혈압 치료법은?" → **이 사용자의 기저질환과 복용약을 고려한** 맞춤 답변 |
| 모든 사람에게 같은 답변 | 사용자마다 다른 답변 |

### 1.2 아키텍처 기반: MedRAG 논문

메디스캐너의 검색 시스템은 **MedRAG 논문** [Xiong et al., 2024]을 기반으로 설계되었습니다.

**MedRAG 논문이란?**

"Benchmarking Retrieval-Augmented Generation for Medicine" (ACL 2024 Findings)는 의료 분야 RAG 시스템의 최적 구성을 실험적으로 검증한 논문입니다. 41가지 조합(코퍼스 × 검색기 × LLM)을 테스트하여 의료 RAG의 모범 사례를 제시했습니다.

**MedRAG 논문의 핵심 발견 (메디스캐너가 채택한 것들)**:

| MedRAG 논문 결론 | 메디스캐너 적용 |
|-----------------|----------------|
| **RRF 융합**이 개별 검색기보다 우수 | ✅ BM25 + 시맨틱 RRF 융합 사용 |
| **BM25**가 의료 도메인에서 여전히 강력 | ✅ BM25를 핵심 검색기로 유지 |
| **다중 코퍼스** 조합이 단일 코퍼스보다 효과적 | ✅ 영어 원천데이터 + 한국어 QA 계획 |
| Top-K 스니펫 수에 따른 log-linear 스케일링 | ✅ Top-5 최적점 확인 |

**MedRAG → 메디스캐너 확장**:

```
┌─────────────────────────────────────────────────────────────────┐
│  MedRAG 논문 (원본)                                              │
├─────────────────────────────────────────────────────────────────┤
│  • 벡터DB: FAISS                                                │
│  • 임베딩: MedCPT (의료 특화, 영어 전용)                          │
│  • 코퍼스: PubMed, StatPearls, Textbooks, Wikipedia (영어)       │
│  • 검색 융합: RRF (BM25 + MedCPT)                                │
└─────────────────────────────────────────────────────────────────┘
                              ↓ 한국어 서비스를 위한 확장
┌─────────────────────────────────────────────────────────────────┐
│  메디스캐너 (확장)                                               │
├─────────────────────────────────────────────────────────────────┤
│  • 벡터DB: FAISS → ChromaDB (경량화, 로컬 실행 용이)              │
│  • 임베딩: MedCPT → multilingual-e5-large (한국어 지원 필수)      │
│  • 코퍼스: 영어 의료문서 + 한국어 QA 데이터 (추가)                 │
│  • 검색 융합: RRF (BM25 + mE5) — 동일 방법론 유지                 │
│  • 추가: 개인 프로필 기반 맞춤 답변                               │
└─────────────────────────────────────────────────────────────────┘
```

**핵심 변경 이유**:

| 변경 사항 | MedRAG 원본 | 메디스캐너 | 이유 |
|----------|-------------|-----------|------|
| **벡터DB** | FAISS | ChromaDB | 경량화, 로컬 실행, 메타데이터 관리 용이 |
| **임베딩 모델** | MedCPT | multilingual-e5-large | **한국어 지원 필수** (MedCPT는 영어 전용) |
| **코퍼스** | 영어 4종 | 영어 + 한국어 | 한국어 고령자 대상 서비스 |

**검색 융합 방법은 MedRAG 논문과 동일**:
- BM25 + Dense Retrieval → RRF 융합
- 논문에서 검증된 k=60 파라미터 사용
- Top-K snippets → LLM 답변 생성

> **📖 참고 논문**: Xiong, G., Jin, Q., Lu, Z., & Zhang, A. (2024). "Benchmarking Retrieval-Augmented Generation for Medicine." ACL 2024 Findings. https://aclanthology.org/2024.findings-acl.372/

> ※ 참고용 정보이니, 정확한 진단은 의료 전문가와 상담하세요."

### 1.4 왜 검색 정확도가 핵심인가

개인 맞춤 답변을 위해서는 **정확한 의료 문서 검색**이 필수입니다.

```
사용자 질문 + 프로필
        ↓
    [검색 시스템] ← 이 단계가 부정확하면?
        ↓
관련 의료 문서 5개 선택
        ↓
    [GPT-4o-mini]
        ↓
    맞춤형 답변
```

**검색이 실패하면**:
- 엉뚱한 문서가 선택됨
- GPT가 잘못된 정보로 답변 생성
- **위험한 의료 정보 제공 가능**

**따라서**: 메디스캐너에서 검색 정확도는 **서비스 품질의 핵심**입니다.

## 3. 세 가지 검색 방법: 쉬운 설명

### 3.1 BM25 (키워드 검색)

**한마디로**: **"글자가 똑같으면 찾는다"**

**비유**: 도서관에서 책 찾기
```
"고혈압" 책 찾아주세요
        ↓
사서가 책 제목/내용에서 "고혈압" 글자를 찾음
        ↓
"고혈압"이 많이 나오는 책일수록 상위 추천
```

**예시**:
| 질문 | 문서 | 결과 |
|------|------|:----:|
| "아스피린 부작용" | "아스피린은 위장에..." | ✅ 찾음 |
| "두통약 부작용" | "아스피린은 위장에..." | ❌ 못 찾음 (두통약≠아스피린) |
| "고혈압" (한국어) | "Hypertension is..." (영어) | ❌ 못 찾음 (언어가 다름) |

**장점**: 정확한 단어(약 이름, 음식 이름) 찾기에 강함
**단점**: 같은 뜻 다른 말, 다른 언어는 못 찾음

---

### 3.2 시맨틱 검색 (의미 검색)

**한마디로**: **"뜻이 비슷하면 찾는다"**

**비유**: 똑똑한 사서에게 물어보기
```
"머리 아플 때 먹는 약" 찾아주세요
        ↓
사서: "아, 진통제 말씀하시는 거죠?"
        ↓
"아스피린", "타이레놀" 관련 책 추천
```

**예시**:
| 질문 | 문서 | 결과 |
|------|------|:----:|
| "두통약 부작용" | "aspirin side effects" | ✅ 찾음 (뜻이 같으니까) |
| "고혈압" (한국어) | "Hypertension is..." (영어) | ✅ 찾음 (의미가 같으니까) |
| "아스피린" | "아세틸살리실산 부작용" | ❌ 놓칠 수 있음 (고유명사 약함) |

**장점**: 다른 언어, 다른 표현도 의미로 찾음
**단점**: 정확한 고유명사(약 이름 등)는 놓칠 수 있음

---

### 3.3 RRF 하이브리드 (둘 다 합치기)

**한마디로**: **"두 방법 결과를 합쳐서 최고를 뽑는다"**

**비유**: 두 사서의 추천을 종합하기
```
질문: "혈압약 먹는데 자몽 먹어도 돼?"

[키워드 사서]              [의미 사서]
1위: 자몽 주의사항         1위: 약물 상호작용 
2위: 약물 상호작용         2위: 혈압약 복용법
3위: 과일 영양소          3위: 식이 주의사항

        ↓ 둘 다 추천한 건? ↓

[최종 결과]
1위: 약물 상호작용 ← 둘 다 추천했으니 이게 정답!
```

**핵심 아이디어**: 두 사서가 **모두 상위권에 올린 문서**가 가장 좋은 문서!

## 4. 세 가지 검색 방법: 전문 설명

### 4.1 BM25 (Best Match 25)

**정의**: 단어의 출현 빈도(TF)와 희소성(IDF)을 기반으로 문서 관련성을 점수화하는 확률적 검색 알고리즘. BEIR 벤치마크(2021)에서 zero-shot 환경에서 강건한 baseline으로 검증되었으며, MedRAG(2024)에서 의료 도메인에서의 효과가 확인됨.

**수식**:
```
score(D, Q) = Σ IDF(qi) × [ f(qi, D) × (k1 + 1) ] / [ f(qi, D) + k1 × (1 - b + b × |D|/avgdl) ]

- Q: 질문 (쿼리)
- D: 문서
- qi: 질문의 i번째 단어
- f(qi, D): 단어 qi가 문서 D에 나타난 빈도
- |D|: 문서 길이
- avgdl: 전체 문서의 평균 길이
- k1: 빈도 포화 상수 (일반적으로 1.2~2.0)
- b: 문서 길이 정규화 파라미터 (일반적으로 0.75)
- IDF(qi): 역문서 빈도 = log((N - n(qi) + 0.5) / (n(qi) + 0.5) + 1)
  - N: 전체 문서 수
  - n(qi): 단어 qi를 포함하는 문서 수
```

**MediScanner 적용**:
- 토크나이징: 한국어는 2글자 이상 단위로 분리
- 라이브러리: `rank_bm25.BM25Okapi`
- 파라미터: k1=1.5, b=0.75 (기본값)

**특성**:
| 장점 | 단점 |
|------|------|
| 정확한 키워드 매칭 (약물명, 고유명사) | 동의어 인식 불가 |
| 계산 속도 빠름 (O(n) 검색) | 언어 간 검색 불가 |
| 해석 가능성 높음 | 의미적 유사성 무시 |

**검증 논문**: 
- Thakur et al. (2021). "BEIR: A Heterogenous Benchmark for Zero-shot Evaluation of IR Models." NeurIPS 2021
- Xiong et al. (2024). "Benchmarking RAG for Medicine." ACL 2024 Findings

### 4.2 시맨틱 검색 (Dense Retrieval)

**정의**: 텍스트를 고차원 벡터(임베딩)로 변환하여 벡터 공간에서의 유사도로 관련성을 측정하는 검색 방법. MedRAG(2024)에서 BM25와 결합하여 의료 도메인 효과 검증.

**작동 원리**:
```
[문서 저장 (오프라인)]
문서 텍스트 → 임베딩 모델 → 1024차원 벡터 → ChromaDB 저장

[검색 (온라인)]
질문 텍스트 → 임베딩 모델 → 1024차원 벡터
        ↓
ChromaDB에서 코사인 유사도 계산
        ↓
가장 유사한 벡터의 문서 반환
```

**코사인 유사도**:
```
cosine_similarity(A, B) = (A · B) / (||A|| × ||B||)

- A, B: 두 벡터
- A · B: 내적 (dot product)
- ||A||: 벡터 A의 크기 (L2 norm)
- 결과: -1 ~ 1 (1에 가까울수록 유사)
```

**MediScanner 적용**:
- 임베딩 모델: `intfloat/multilingual-e5-large` (1024차원)
- 벡터DB: ChromaDB
- E5 Prefix 규칙:
  - 문서 저장 시: `"passage: " + 문서`
  - 검색 쿼리 시: `"query: " + 질문`

**Cross-lingual Retrieval**:
```
multilingual-e5 모델의 특성:
- 100개 이상 언어 지원
- 같은 의미의 텍스트는 언어가 달라도 벡터 공간에서 가까이 위치

예시:
"고혈압"     → [0.23, -0.15, 0.82, ...]
"hypertension" → [0.25, -0.12, 0.79, ...]
  → 코사인 유사도 높음 → 검색 성공!
```

**MedCPT 대신 multilingual-e5를 선택한 이유**:

MedRAG 논문에서 MedCPT가 의료 도메인에서 가장 높은 성능을 보였지만, MedCPT는 **영어만 지원**합니다. 메디스캐너는 한국어 질문을 처리해야 하므로 **다국어 지원**이 필수이며, 이를 위해 multilingual-e5-large를 선택했습니다.

| 모델 | 언어 지원 | 의료 특화 | 메디스캐너 적합성 |
|------|:--------:|:--------:|:---------------:|
| MedCPT | 영어만 | ✅ | ❌ 한국어 불가 |
| multilingual-e5 | 100개 언어 | ❌ | ✅ 한국어↔영어 |

**특성**:
| 장점 | 단점 |
|------|------|
| 의미적 유사도 파악 | 계산 비용 높음 (GPU 권장) |
| Cross-lingual 검색 가능 | 고유명사/희귀 단어에 약함 |
| 동의어, 유사 표현 인식 | "시맨틱 유사 함정" (관련 없는데 유사) |

**검증 논문**: Wang, L. et al. (2024). "Multilingual E5 Text Embeddings: A Technical Report." arXiv:2402.05672

### 4.3 RRF (Reciprocal Rank Fusion)

**정의**: 여러 검색 방법의 결과를 순위 기반으로 융합하는 알고리즘. 점수의 스케일이 다른 검색 방법들을 비교 가능하게 통합. MedRAG(2024)에서 BM25 + Dense Retrieval 융합에 사용되어 효과 검증됨.

**RRF 수식**:
```
RRF_score(d) = Σ 1 / (k + rank_i(d))

- d: 문서
- k: 상수 (일반적으로 60)
- rank_i(d): i번째 검색 방법에서 문서 d의 순위 (1부터 시작)
- Σ: 모든 검색 방법에 대해 합산
```

**예시 계산**:
```
질문: "혈압약 먹는데 자몽 먹어도 돼?"

문서 A:
- 시맨틱 검색 순위: 1위 → 1/(60+1) = 0.0164
- BM25 검색 순위:   2위 → 1/(60+2) = 0.0161
- RRF 점수: 0.0164 + 0.0161 = 0.0325

문서 B:
- 시맨틱 검색 순위: 3위 → 1/(60+3) = 0.0159
- BM25 검색 순위:   1위 → 1/(60+1) = 0.0164
- RRF 점수: 0.0159 + 0.0164 = 0.0323

결과: 문서 A(0.0325) > 문서 B(0.0323) → 문서 A가 1위
```

**MediScanner 적용**:
```python
def rrf_fusion(semantic_docs, bm25_docs, k=60):
    rrf_scores = {}
    
    # 시맨틱 검색 결과 순위 반영
    for rank, doc in enumerate(semantic_docs):
        doc_id = doc["id"]
        rrf_scores[doc_id] = rrf_scores.get(doc_id, 0) + 1.0 / (k + rank + 1)
    
    # BM25 검색 결과 순위 반영
    for rank, doc in enumerate(bm25_docs):
        doc_id = doc["id"]
        rrf_scores[doc_id] = rrf_scores.get(doc_id, 0) + 1.0 / (k + rank + 1)
    
    # 점수 기준 정렬
    sorted_docs = sorted(rrf_scores.items(), key=lambda x: x[1], reverse=True)
    return sorted_docs
```

**핵심 아이디어**:
1. **점수가 아닌 순위 사용**: 서로 다른 스케일의 점수를 비교할 필요 없음
2. **상호 보완**: 한 방법에서 낮은 순위여도 다른 방법에서 높으면 최종 순위 상승
3. **k=60**: 상위 문서에 더 큰 가중치 부여 (MedRAG 등 다수 연구에서 검증된 표준값)

**특성**:
| 장점 | 단점 |
|------|------|
| 두 검색 방법의 장점 결합 | 둘 다 실패하면 보완 불가 |
| 점수 정규화 불필요 | 추가 계산 비용 (미미함) |
| 튜닝 파라미터 적음 (k만 조절) | |

**검증 논문**: Xiong et al. (2024). "Benchmarking RAG for Medicine." ACL 2024 Findings - BM25 + MedCPT 융합에 RRF 적용하여 최적 검색 구성 도출

### 5.4 Top-K 완전 가이드

#### 5.4.1 Top-K란?

**Top-K의 의미**: 검색 결과 **상위 K개 청크**를 GPT에게 전달하는 것

**"문서"가 아니라 "청크"**:
```
원본 영어 문서 (긴 글, 예: 3000자)
        ↓
800자씩 잘라서 저장 (청킹)
        ↓
총 409,741개 청크가 ChromaDB에 저장됨
        ↓
Top-5 = 상위 5개 "청크" 선택
```

- 원본 문서가 3000자면 → 4개 청크로 분할됨
- 검색 결과는 "문서" 단위가 아니라 **"청크" 단위**

---

#### 5.4.2 랭킹은 어떻게 매겨지나?

**① BM25: 키워드 매칭 점수**

```
질문: "자몽 혈압약 상호작용"
        ↓
40만 청크 각각에 BM25 점수 계산
        ↓
점수 = "자몽" 등장 횟수 × 희귀도 + "혈압약" × 희귀도 + ...
        ↓
점수 높은 순으로 정렬 → 1위, 2위, 3위...
```

| 청크 | "자몽" 등장 | "혈압약" 등장 | BM25 점수 | 순위 |
|:----:|:---------:|:-----------:|:---------:|:----:|
| A | 3번 | 2번 | 8.5 | 1위 |
| B | 1번 | 3번 | 7.2 | 2위 |
| C | 0번 | 5번 | 4.1 | 3위 |

**② 시맨틱: 코사인 유사도 점수**

```
질문: "자몽 혈압약 상호작용"
        ↓
질문을 벡터로 변환: [0.23, -0.15, 0.82, ...]  (1024차원)
        ↓
40만 청크 각각의 벡터와 코사인 유사도 계산
        ↓
유사도 높은 순으로 정렬 → 1위, 2위, 3위...
```

| 청크 | 코사인 유사도 | 순위 |
|:----:|:-----------:|:----:|
| X | 0.95 | 1위 |
| Y | 0.89 | 2위 |
| Z | 0.82 | 3위 |

**③ RRF: 두 순위를 융합**

```
BM25 결과:    1위-A, 2위-B, 3위-C, 4위-X, 5위-Y
시맨틱 결과:  1위-X, 2위-Y, 3위-A, 4위-Z, 5위-B

        ↓ RRF 점수 계산 (k=60) ↓

청크 A: 1/(60+1) + 1/(60+3) = 0.0164 + 0.0159 = 0.0323
청크 X: 1/(60+4) + 1/(60+1) = 0.0156 + 0.0164 = 0.0320

        ↓ RRF 점수 순으로 정렬 ↓

RRF 최종: 1위-A, 2위-X, 3위-B, ...
```

**랭킹 기준 요약**:

| 검색 방법 | 랭킹 기준 | 점수 의미 |
|----------|----------|----------|
| **BM25** | 키워드 매칭 점수 | 단어가 많이/희귀하게 등장할수록 높음 |
| **시맨틱** | 코사인 유사도 | 의미가 비슷할수록 1에 가까움 |
| **RRF** | 두 순위의 융합 | 둘 다 상위권이면 높음 |

| **출력 길이** | 프롬프트로 제어 | 프롬프트로 제어 |

| 문제 | 설명 |
|------|------|
| **노이즈 증가** | 6~10위 청크가 관련 없으면 GPT가 혼란 |

#### 5.4.5 Top-K Hit Rate란?

**평가에서 사용하는 지표**:
- "상위 K개 청크 중 **정답 키워드가 포함된 청크가 있는가?**"
- 예: Top-5 Hit Rate 96% = 100개 질문 중 96개는 상위 5개 안에 정답 청크 존재

### 6.1 시맨틱(92%) > BM25(83%)인 이유

**핵심 원인: Cross-lingual 상황**

```
현재 데이터 구조:

- 문서: 영어 (CDC, WHO, Medscape)

질문: "고혈압 치료 방법은?"
  → 벡터: [0.23, -0.15, 0.82, ...]

  → ✅ 매칭 성공 (의미가 같으므로 벡터가 가까움)

### 6.2 BM25의 Top-5 → Top-10 큰 개선 (+11%p)의 의미

- 정확한 키워드 매칭 시 높은 점수
- 하지만 Cross-lingual 환경에서 정확도 저하
- 정답 문서가 6~10위에 밀려 있는 경우가 많음

예시:
질문: "당뇨병 합병증"

  1~5위: 다른 키워드가 우연히 매칭된 문서

```

**시사점**: BM25 단독 사용 시 Top-K를 크게 잡아야 하지만, RRF 융합 시 불필요

### 6.5 증상-질병에서 가장 낮은 정확도 (85%)인 이유

**실패한 질문 예시**:
```
질문: "요즘 자꾸 목이 마르고 화장실을 자주 가는데 무슨 병이야?"

사용자 표현      →    의학 용어        →    영어 문서
─────────────────────────────────────────────────────
"목이 마르다"    →    "다갈증, 갈증"   →    "polydipsia"
"화장실 자주"    →    "빈뇨, 다뇨"     →    "polyuria"
```

**이중 장벽**:
1. **언어 장벽**: 한국어 질문 ↔ 영어 문서
2. **용어 장벽**: 일상 구어체 ↔ 의학 전문 용어

**문제점**:
- multilingual-e5는 일반 다국어 모델 (의료 특화 아님)
- "목이 마르다" → "polydipsia" 연결은 의료 도메인 지식 필요
- 현재 모델로는 추론 한계

**개선 방안**:
- 한국어 QA 데이터 추가로 직접적인 증상-질병 매핑 강화

### 6.6 RRF 효과 분석 (+4%p)

**상호 보완 분석 (100개 기준)**:

| 케이스 | 시맨틱 | BM25 | RRF | 발생 비율 |
|--------|:------:|:----:|:---:|:---------:|
| 둘 다 성공 | ✅ | ✅ | ✅ | 79% |
| 시맨틱만 성공 | ✅ | ❌ | ✅ | 13% |
| BM25만 성공 | ❌ | ✅ | ✅ | 4% |
| 둘 다 실패 | ❌ | ❌ | ❌ | 4% |

**RRF Top-5 = 96% 달성 원리**:

- 둘 다 실패하는 4%만 해결 불가

**결론**: RRF는 효과적이나, **둘 다 실패하는 케이스(4%)는 다른 방법 필요**

## 7. LLM 모델 선택: GPT-4o-mini

### 7.1 GPT-4o-mini API란?

**GPT-4o-mini**는 OpenAI에서 제공하는 대규모 언어 모델(LLM) API입니다.

**쉬운 설명**:
```
GPT-4o-mini = "똑똑한 AI 비서"

- 텍스트를 읽고 이해함
- 질문에 대한 답변을 생성함
- 영어 문서를 한국어로 번역/요약함
- API로 호출하면 답변을 받을 수 있음 (유료)
```

**API(Application Programming Interface)란?**:
```
우리 서비스 (메디스캐너)
        ↓
   API 호출 (인터넷으로 요청)
        ↓
OpenAI 서버 (GPT-4o-mini)
        ↓
   답변 반환
        ↓
우리 서비스에서 사용자에게 표시
```

- 우리가 직접 AI 모델을 만들 필요 없음
- OpenAI의 모델을 "빌려서" 사용
- 사용한 만큼 비용 지불 (토큰 단위 과금)

---

### 7.2 메디스캐너에서 GPT-4o-mini의 역할

        ↓ GPT-4o-mini 처리 ↓

[한국어 답변]
"자몽은 고혈압약(칼슘채널차단제, 예: 암로디핀)과

### 7.4 결론: LLM 선택 전략

```
┌────────────────────────────────────────────────────────────┐
│  메디스캐너 = 검색(RAG) + 생성(LLM)                          │
│                                                              │
│  검색 품질이 핵심!                                           │
│  - 검색이 정확하면 → 간단한 LLM으로도 좋은 답변              │
│  - 검색이 실패하면 → 아무리 좋은 LLM도 소용없음              │
│                                                              │
│  따라서:                                                     │
│  - 검색 시스템에 투자 (RRF, 임베딩 최적화)                   │
│  - LLM은 가성비 모델 선택 (GPT-4o-mini)                      │
└────────────────────────────────────────────────────────────┘
```

| 질문 | 답변 |
|------|------|
| 왜 GPT-4o-mini? | **가성비 최고** (GPT-4o 대비 30배 저렴) |
| 성능은 충분해? | **RAG 기반이라 충분** (요약/번역 역할) |
| 더 좋은 모델 필요? | 검색 품질이 좋으면 **불필요** |

## 8. 향후 실험 계획

### 8.1 한국어 QA 데이터 임베딩 및 테스트

**데이터**: 라벨링데이터 (31,036개 한국어 QA)
- 1_필수의료: 17,280개 (Training 15,360 + Validation 1,920)
- 2_전문의학: 13,756개 (Training 12,228 + Validation 1,528)

**임베딩 모델**: `intfloat/multilingual-e5-large` (1024차원) — 원천데이터와 동일

**예상 효과**:
- 한국어 질문 ↔ 한국어 문서 직접 매칭
- 증상-질병 카테고리 정확도 향상 기대 (현재 85% → 목표 95%+)
- BM25 성능 향상 기대 (같은 언어)

**테스트 계획**:
1. 한국어 QA 데이터만으로 테스트
2. 동일한 100개 어려운 질문으로 평가
3. 카테고리별 비교

### 8.2 원천데이터 + 한국어 QA 통합 테스트

**검색 흐름**:
```
질문 (한국어)
        ↓
[1단계] 한국어 QA 데이터 검색 (RRF)
        ↓
[2단계] 원천데이터 검색 (RRF)
        ↓
두 결과 통합 (가중치 조절)
        ↓
Top-5 최종 선정
```

**테스트 항목**:
- 통합 검색의 정확도
- 한국어 QA / 원천데이터 결과 가중치 최적화
- 검색 속도 측정

### 8.3 최종 목표

| 지표 | 현재 (원천데이터만) | 목표 (통합) |
|------|:------------------:|:----------:|
| RRF Top-5 Hit Rate | 96.0% | **98%+** |
| 증상-질병 카테고리 | 85.0% | **95%+** |
| 실패 케이스 | 4개 | **0~1개** |

## 9. 요약

### 9.1 핵심 발견

1. **Cross-lingual 환경에서 시맨틱 > BM25**
   - multilingual-e5가 언어 장벽 극복
   - 시맨틱 92.0% > BM25 83.0%

2. **RRF 하이브리드가 가장 효과적**
   - RRF Top-5 = 96% (시맨틱 대비 +4%p)
   - Top-5만으로 최대 성능 달성 (Top-10 불필요)

3. **RRF의 순위 끌어올림 효과**
   - BM25는 Top-5→Top-10에서 +11%p 개선 필요
   - RRF는 Top-5에서 이미 최대치 → **토큰 비용 50% 절감**

4. **증상-질병 질문은 여전히 어려움**
   - RRF에서도 85% (가장 낮은 카테고리)
   - 구어체→의학용어 추론 한계
   - 한국어 QA 데이터 추가로 개선 기대

5. **고유명사(약물/음식)는 BM25가 강점**
   - "시금치", "자몽" 등 키워드 매칭
   - RRF에서 BM25가 시맨틱 보완

### 9.2 현재 한계

- 원천데이터만 테스트한 결과 (영어 ~90% + 한국어 ~10%)
- 실제 서비스는 한국어 QA 우선 검색 필요
- 한국어 QA 데이터(31,036건) 임베딩 후 재테스트 필요

### 9.3 다음 단계

1. ⏳ 한국어 QA 데이터 임베딩 (31,036건)
2. ⏳ 한국어 QA 데이터만 성능 테스트
3. ⏳ 원천데이터 + 한국어 QA 통합 성능 테스트
4. ⏳ 최종 검색 파이프라인 확정

### 1. MedRAG (핵심 기반)

**Xiong, G., Jin, Q., Lu, Z., & Zhang, A. (2024)** - "Benchmarking Retrieval-Augmented Generation for Medicine." *ACL 2024 Findings*, pages 6233-6251. https://aclanthology.org/2024.findings-acl.372/

- **메디스캐너 아키텍처의 핵심 기반**
- BM25 + Dense Retrieval + RRF 융합 방법론 검증
- 41가지 조합(코퍼스 × 검색기 × LLM) 실험으로 최적 구성 도출
- GPT-3.5 정확도를 최대 18% 향상시켜 GPT-4 수준 달성

### 2. Multilingual E5 (임베딩 모델)

**Wang, L., Yang, N., Huang, X., Yang, L., Majumder, R., & Wei, F. (2024)** - "Multilingual E5 Text Embeddings: A Technical Report." *arXiv:2402.05672*. https://arxiv.org/abs/2402.05672

- 메디스캐너에서 사용하는 `intfloat/multilingual-e5-large` 모델
- 100개+ 언어 지원, 1024차원 벡터
- 10억 개 다국어 텍스트 쌍으로 contrastive pre-training

### 3. BEIR (검색 벤치마크)

**Thakur, N., Reimers, N., Rücklé, A., Srivastava, A., & Gurevych, I. (2021)** - "BEIR: A Heterogenous Benchmark for Zero-shot Evaluation of Information Retrieval Models." *NeurIPS 2021 Datasets and Benchmarks Track*. https://arxiv.org/abs/2104.08663

- 18개 데이터셋, 10개 검색 모델 비교 평가
- **BM25가 zero-shot 환경에서 강건한 baseline임을 검증**
- Dense retrieval이 in-domain에서 우수하나 OOD 일반화에 한계
